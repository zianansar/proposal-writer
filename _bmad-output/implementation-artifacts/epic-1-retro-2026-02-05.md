### 1. Race Conditions in Async Operations (Recurring Pattern)

**Evidence:**
- Story 1-13: Race in retry logic (incrementRetry() timing)
- Story 1-14: Race in draft auto-save (concurrent saves violating NFR-11)

**Impact:**
- Required fixes and re-testing
- Pattern suggests async Rust complexity underestimated
- Similar issues likely in Epic 2 if not addressed

**Root Cause:** Async operations with shared state (Mutex, channels) require careful ordering and synchronization.

**Action Required:** Create async testing guidelines and patterns document.

**Evidence:** Story 1-16 log level initialization order conflict.

**Impact:**

**Root Cause:** Initialization order not planned upfront - discovered during implementation.

**Action Required:** Add initialization order checklist to architecture review.

## Key Insights & Lessons Learned ðŸ’¡

### 1. Separate Code Review Agent Finds More Issues

**Insight:** Using a separate "adversarial reviewer" with fresh context found 3-10 issues per story, including critical race conditions and missing integrations that dev agent missed.

**Evidence:**
- Story 1-13: 2 HIGH issues found (race conditions)
- Story 1-14: 3 HIGH issues found (including NFR violations)
- Story 1-16: 13 issues found (8 HIGH, 3 MEDIUM, 2 LOW)

**Application:** Continue code review workflow for all stories in Epic 2. Consider increasing review frequency for complex/risky stories (migration, encryption).

### 2. Initialization Order Requires Upfront Planning

**Insight:** Setup sequence (logging vs DB vs config) has interdependencies that must be designed, not discovered.

**Evidence:** Story 1-16 discovered logging needs DB for settings, but DB errors need logging - resulted in deferred work.

**Application:** Before Epic 2 starts, design complete initialization sequence:
- What order: passphrase entry â†’ keychain retrieval â†’ logging init â†’ DB migration â†’ config load
- Document dependencies and constraints
- Validate in architecture review before implementation

### 3. Async Testing Needs Different Patterns

**Insight:** Rust async operations with shared state are difficult to test correctly. Race conditions found in code review, not during testing.

**Evidence:** Stories 1-13 and 1-14 both had race conditions that passed tests but were caught in review.

**Application:**
- Create async testing patterns guide (channel draining, tokio::time::pause, etc.)
- Add concurrency tests for shared state operations
- Review async code extra carefully

### 4. Manual Test Steps Need Explicit Checklists

**Insight:** "Manual verification required" tasks are consistently deferred/skipped without explicit checklists.

**Evidence:** Story 1-1 NFR validation, Story 0-5 AI detection testing.

**Application:**
- Add "Manual Test Checklist" section to story template
- Require sign-off from QA or user before marking done
- Consider which manual tests can be automated

### 5. Deferred Debt Needs Epic-Level Visibility

**Insight:** Debt documented in individual stories is easy to lose track of. Epic-level summary makes debt visible.

**Evidence:** Story 1-16 created epic-2-deferred-tasks.md to track log level initialization task.

**Application:**
- At end of each epic, create summary document for deferred work
- Include in next epic's planning
- Prevents "forgotten debt" accumulation
